       - `PaymentSettings`: card_number, sbp_phone, bank_name, has_robokassa (bool, computed), robokassa_is_test, robokassa_hash_algorithm, fiscalization_level, has_seen_grey_warning, receipt_sno. `model_config = {"from_attributes": True}`.
       - `PaymentSettingsUpdate`: card_number, sbp_phone, bank_name, fiscalization_level, receipt_sno (all optional).
       - `RobokassaSetup`: merchant_login (str, min_length=1), password1 (str, min_length=1), password2 (str, min_length=1), is_test (bool = False), hash_algorithm (str, pattern="^(md5|sha256|sha512)$", default="sha256").
       - `RobokassaDisconnect`: Empty model (just confirmation).

    3. **Create PaymentService** (backend/app/services/payment_service.py, ~250 lines):
       - `create_manual_payment(db, master, booking_id, payment_method, fiscalization_override)`:
         - Validate booking belongs to master, status is "confirmed"
         - Create Payment with status="paid", paid_at=now, amount from booking.service.price
         - Update booking status to "completed"
         - Determine effective fiscalization level (override or master default)
         - If fiscalization is "manual": set receipt_data as JSON with formatted data (amount, service_name, client_name, date)
         - Return Payment

       - `create_robokassa_payment(db, master, booking_id, fiscalization_override)`:
         - Validate booking, decrypt master's Robokassa credentials
         - Get next InvId from sequence: `SELECT nextval('robokassa_inv_id_seq')`
         - Build receipt JSON if fiscalization is "auto" (using `RobokassaService.build_receipt_json`)
         - Generate payment URL with `Shp_master_id` and `Shp_booking_id` params
         - Create Payment with status="pending", robokassa_inv_id=inv_id, payment_url=url
         - Return Payment (caller sends link to client)

       - `create_requisites_payment(db, master, booking_id, fiscalization_override)`:
         - Validate booking, get master's card_number/sbp_phone
         - Generate QR code from sbp_phone (preferred) or card_number
         - Create Payment with status="pending"
         - Return Payment + requisites data (card, phone, qr_base64)

       - `confirm_requisites_payment(db, master, payment_id, payment_method)`:
         - Find payment, validate status="pending", set to "paid", paid_at=now, payment_method
         - Update booking to "completed"
         - Handle fiscalization same as manual

       - `handle_robokassa_callback(db, master_id, inv_id, out_sum, signature, shp_params)`:
         - Look up payment by robokassa_inv_id
         - Idempotency: if already "paid", return True (success, skip processing)
         - Decrypt master's credentials, verify signature via RobokassaService
         - If valid: update status to "paid", paid_at=now, update booking to "completed"
         - If auto fiscalization: set receipt_status="issued" (Robochecks handles it via Receipt param)
         - Return True/False

       - `get_payment_history(db, master_id, filters)`:
         - Query payments with master_id, apply status/date filters, order by created_at desc
         - Join with booking -> service, booking -> client for display names
         - Return paginated list

       - `cancel_payment(db, master, payment_id)`:
         - Find payment, set status="cancelled"
         - If receipt_status="issued" and Robokassa connected: set receipt_status="cancelled" (receipt cancellation is reminder-only in v1, no API call)
         - Return Payment

       - Static helper: `format_receipt_data(amount_kopecks, service_name, client_name) -> dict` for manual receipt copy data

    Use `async with db.begin():` pattern consistent with existing services. Use SELECT FOR UPDATE when modifying payments to prevent race conditions.
