# МоиОкошки v2.0 — Design Spec

## Overview

Расширение CRM до полноценной платформы для самозанятых мастеров с публичной страницей записи, кабинетом клиента, автоматическими чеками в ФНС и аналитикой. Основан на анализе YCLIENTS и их слабых мест.

**Core differentiators vs YCLIENTS:**
- Бесплатно для solo-мастеров (YCLIENTS от 686₽/мес)
- Нативные TG/MAX/VK боты (YCLIENTS — только через третьи стороны)
- Публичная профильная страница мастера с портфолио (YCLIENTS не имеет)
- Автоматический чек в "Мой Налог" для самозанятых через API ФНС (уникальная фича)
- Modern mobile-first UI

## 1. Публичная страница мастера

### URL
`moiokoshki.ru/m/{username}` — мастер задаёт username в настройках профиля. Fallback на UUID если username не задан: `moiokoshki.ru/m/{master_uuid}`.

### Структура страницы (mobile-first, сверху вниз)

**Hero-секция:**
- Аватар мастера (круг, 96px)
- Имя, специализация (одна строка, напр. "Мастер маникюра")
- Город
- Рейтинг: звёзды (1-5) + кол-во отзывов ("4.8 ★ · 24 отзыва")
- Кнопка "Записаться" (primary, full-width)

**Портфолио (галерея):**
- Горизонтальный scroll фото работ (до 30 фото)
- Lightbox при клике (swipe между фото)
- Опциональные теги по услугам

**Услуги:**
- Карточки: название, цена (₽), длительность (мин)
- Кнопка "Записаться" на каждой карточке → переход в flow записи с предвыбранной услугой

**Ближайшие окошки:**
- 3-5 дней вперёд, показать первые свободные слоты
- Клик → переход в flow записи с предвыбранным временем

**Отзывы:**
- Последние 10 отзывов (имя клиента, дата, оценка 1-5, текст)
- Средний рейтинг вверху секции
- Кнопка "Все отзывы" → scroll к полному списку

**Контакты:**
- Ссылки на мессенджеры мастера (TG/VK/Instagram)
- Адрес если указан

**Sticky footer:**
- Фиксированная кнопка "Записаться" (visible on scroll)

### Flow записи (step-by-step)

1. **Услуга** (если не предвыбрана из карточки)
2. **Дата** → календарь с доступными днями
3. **Время** → слоты на выбранный день
4. **Телефон + имя** клиента
5. **Подтверждение** → summary card, кнопка "Записаться"

После записи: клиент видит подтверждение + ссылку на кабинет ("Следить за записями").

### Шаринг

- В настройках мастера: кнопка "Скопировать ссылку" + автогенерация QR-кода
- Мастер вставляет ссылку в Instagram bio, визитку, WhatsApp статус, VK

### Технология

Отдельный React SPA в директории `/public/` с отдельным vite.config (base: "/m/"). Caddy роут: `/m/*` → public:3002. Без авторизации — полностью публичная страница.

Backend: новые public endpoints:
- `GET /api/v1/masters/{username}/profile` — профиль, портфолио, рейтинг
- `GET /api/v1/masters/{username}/services` — услуги
- `GET /api/v1/masters/{username}/slots?date=YYYY-MM-DD` — слоты
- `GET /api/v1/masters/{username}/reviews` — отзывы
- `POST /api/v1/bookings` — создание записи (существует)

## 2. Кабинет клиента

### Доступ

URL: `moiokoshki.ru/my`

Авторизация: клиент вводит телефон → получает 4-значный OTP код в мессенджер (TG/MAX/VK бот, через который ранее записывался). Сессия хранится в cookie (7 дней).

Fallback: если клиент записывался через веб (без мессенджера) и нет match по телефону ни в одном боте → показываем "Свяжитесь с мастером для подтверждения".

### Экраны

**Мои записи (главный экран):**

Предстоящие:
- Карточки: аватар мастера, имя мастера, услуга, дата/время, адрес
- Кнопки: "Отменить" (если deadline не прошёл), "Перенести" (→ выбор нового времени)

Прошедшие:
- Те же карточки + кнопки "Записаться снова" и "Оставить отзыв"
- "Записаться снова" → flow записи с предвыбранным мастером + услугой, только выбор даты/времени

Если клиент ходит к нескольким мастерам — все видны в одном списке, сгруппированы по дате.

**Клик на имя мастера:**
- Переход на публичную страницу мастера `/m/{username}`

### Технология

Тот же SPA что и публичная страница (`/public/`), роут `/my`. Авторизация по телефону + OTP.

Backend:
- `POST /api/v1/client/auth/request-code` — отправить OTP
- `POST /api/v1/client/auth/verify-code` — проверить OTP, вернуть session token
- `GET /api/v1/client/bookings` — записи клиента (по телефону, все мастера)
- `POST /api/v1/client/reviews` — оставить отзыв

## 3. Платежи в админке

### Flow завершения визита

В BookingDrawer (веб-админка) на подтверждённой записи → кнопка "Завершить визит" → модальное окно:

**Модальное окно "Приём оплаты":**
- **Сумма:** предзаполнена из стоимости услуги, редактируемое поле (скидка/доплата)
- **Способ оплаты:** pill-selector:
  - Наличные
  - Перевод на карту
  - СБП (Robokassa) — если подключена
- **Чек в ФНС:** pill-selector (если ИНН привязан):
  - Автоматически (чек создаётся в "Мой Налог")
  - Вручную (CRM покажет данные для ручного ввода)
  - Без чека (с предупреждением)
- Кнопка "Завершить и принять оплату"

**Результат:**
- Запись → статус "completed"
- Платёж создаётся с выбранным методом
- Если СБП → клиент получает ссылку на оплату в мессенджер
- Если наличные/перевод → платёж сразу "paid"
- Если авточек → чек отправляется в ФНС → клиент получает ссылку на чек

**Расширение страницы "Платежи":**
- Фильтр по способу оплаты (наличные/карта/СБП)
- Итого выручка за период (сумма paid)
- Статус чека (выдан/не выдан/ошибка)
- Экспорт в CSV

### Быстрая оплата из календаря
Иконка ₽ на карточке записи в календаре → открывает BookingDrawer с фокусом на оплате.

## 4. Автоматический чек в ФНС для самозанятых

### Привязка ИНН (в настройках мастера)

**Flow:**
1. Мастер → Настройки → "Налоги и чеки" → "Подключить Мой Налог"
2. Вводит ИНН (12 цифр)
3. Система проверяет статус самозанятого через API ФНС (`/api/v1/taxpayer/status`)
4. Если ОК → мастер получает SMS от ФНС с кодом подтверждения
5. Вводит код → привязка завершена
6. В настройках показывается: "Мой Налог подключён ✓", ИНН, кнопка "Отключить"

### Автоформирование чека

**Триггер:** Мастер завершает визит и выбирает "Чек: Автоматически"

**Что отправляется в ФНС:**
- Название услуги (из booking → service)
- Сумма (из payment)
- Дата (из booking.starts_at)
- Данные клиента: имя (из client)
- ИНН мастера (из настроек)

**Результат:**
- ФНС возвращает ссылку на чек (receipt_url)
- Ссылка сохраняется в payment record
- Клиент получает ссылку на чек в мессенджер
- В платежах отображается статус чека: "Выдан ✓" со ссылкой

### API ФНС "Мой Налог"

Используем партнёрский API lknpd.nalog.ru:
- `POST /income` — создать доход (выдать чек)
- `DELETE /income/{receipt_id}` — аннулировать чек (при отмене/возврате)
- `GET /taxpayer/status` — проверить статус самозанятого

Credentials: ИНН мастера + токен авторизации (получается через OAuth flow ФНС или через ввод пароля от ЛК налогоплательщика). Токен хранится зашифрованным в БД (аналогично Robokassa credentials).

### Fallback (при ошибке API ФНС)

Если API ФНС недоступен:
- Платёж создаётся, визит завершается
- Чек помечается "pending"
- Background job (APScheduler) ретраит каждые 5 минут, до 3 попыток
- Если не удалось → чек помечается "error", мастер получает уведомление "Чек не был выдан, оформите вручную"

## 5. Портфолио мастера

### Загрузка фото

**Где:** Настройки мастера → "Мои работы" (и в админке, и в мини-аппе)

**Ограничения:**
- До 30 фото
- Макс 5MB на фото
- Форматы: JPEG, PNG, WebP
- Автоматическое сжатие до 1200px по длинной стороне
- Генерация thumbnail (300px) для галереи

**Хранение:**
- Docker volume `/data/portfolio/` (для MVP)
- Миграция на S3/MinIO позже при необходимости

**Теги:**
- Мастер может присвоить фото тег с названием услуги
- На публичной странице — фильтр по тегам

### Модель данных

Новая таблица `portfolio_photos`:
- id (UUID)
- master_id (FK → masters)
- file_path (varchar)
- thumbnail_path (varchar)
- caption (varchar, nullable)
- service_tag (varchar, nullable)
- sort_order (int)
- created_at (timestamp)

## 6. Отзывы

### Автоматический сбор

**Триггер:** Через 2 часа после `ends_at` записи со статусом "completed" → бот отправляет клиенту:

> "Спасибо за визит к {мастер}! Как всё прошло? Оцените от 1 до 5 ⭐"
> [1⭐] [2⭐] [3⭐] [4⭐] [5⭐]

Клиент нажимает звезду → бот спрашивает:
> "Хотите добавить комментарий?" → [Да, написать] [Нет, спасибо]

Если да → клиент пишет текст (до 500 символов).

### Модерация

- Отзывы ≥3 звёзд → публикуются автоматически
- Отзывы <3 звёзд → мастер видит в "Отзывы" в админке, может ответить
- Публикуется только после ответа мастера (мотивирует работать с негативом)
- Мастер не может удалять отзывы (только скрывать с указанием причины для модерации)

### Защита от накрутки

- Отзыв привязан к booking_id (1 отзыв на 1 визит)
- Только завершённые визиты ("completed")
- Нельзя оставить отзыв если прошло >30 дней после визита

### Модель данных

Новая таблица `reviews`:
- id (UUID)
- booking_id (FK → bookings, unique)
- master_id (FK → masters)
- client_id (FK → clients)
- rating (int, 1-5)
- text (varchar(500), nullable)
- master_reply (varchar(500), nullable)
- status (varchar: "published", "pending_reply", "hidden")
- created_at (timestamp)

## 7. Аналитика

### Расположение

Веб-панель мастера → новый пункт sidebar "Аналитика" (иконка BarChartOutlined).

### Дашборд (Tab 1: "Обзор")

**Период:** переключатель Сегодня / Неделя / Месяц / Произвольный (RangePicker)

**3 карточки-метрики (верхний ряд):**
- Доход (₽) — сумма paid платежей за период
- Записей — кол-во bookings за период
- Клиентов — уникальных клиентов за период

**График дохода:**
- Линейный график по дням за выбранный период (Ant Design Charts или recharts)

**Новые vs повторные клиенты:**
- Donut chart: % новых (первый визит) vs повторных

**Загруженность:**
- Progress bar: (сумма duration_minutes всех completed bookings) / (рабочие часы по расписанию за период)

**Средний чек:**
- Доход / кол-во оплаченных визитов

### Детальный отчёт (Tab 2: "Отчёты")

**Топ услуг (таблица):**
- Услуга, кол-во записей, доход (₽), % от общего дохода
- Сортировка по доходу

**По дням (таблица):**
- Дата, кол-во записей, доход, загруженность %

**Экспорт:** кнопка "Скачать CSV"

### Backend

Новые endpoints:
- `GET /api/v1/analytics/summary?from=&to=` — метрики за период
- `GET /api/v1/analytics/revenue-chart?from=&to=` — данные для графика
- `GET /api/v1/analytics/top-services?from=&to=` — топ услуг
- `GET /api/v1/analytics/daily?from=&to=` — по дням

Данные берутся из `payments` + `bookings` + `master_clients`. Прямые SQL-запросы с агрегацией — для solo-мастера объём данных маленький, отдельные таблицы агрегации не нужны.

## 8. Изменения в существующей системе

### Модель Master (расширение)

Новые поля:
- `username` (varchar, unique, nullable) — для URL `/m/{username}`
- `specialization` (varchar, nullable) — "Мастер маникюра"
- `city` (varchar, nullable)
- `avatar_path` (varchar, nullable)
- `inn` (varchar(12), nullable) — ИНН для самозанятых
- `fns_token_encrypted` (text, nullable) — токен API ФНС (Fernet encrypted)
- `fns_connected` (boolean, default false)
- `instagram_url` (varchar, nullable)

### Модель Payment (расширение)

Новое поле:
- `fns_receipt_url` (varchar, nullable) — ссылка на чек из ФНС

### Новые таблицы

- `portfolio_photos` (см. секцию 5)
- `reviews` (см. секцию 6)
- `client_sessions` — сессии кабинета клиента (id, client_id, phone, token, expires_at)

### Caddy (новый роут)

```
handle /m/* {
    uri strip_prefix /m
    reverse_proxy public:3002
}

redir /m /m/ 308

handle /my/* {
    uri strip_prefix /my
    reverse_proxy public:3002
}

redir /my /my/ 308
```

### Docker Compose (новый сервис)

```yaml
public:
  build:
    context: ./public
    dockerfile: Dockerfile
  ports:
    - "3002:3002"
  restart: unless-stopped
```
